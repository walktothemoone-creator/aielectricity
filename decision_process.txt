[ 설계 철학 ]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  전기수요 예측은 단일 모델로 해결되지 않는다. 수요는 날씨(단기 변동), 산업구조
  (중기 구조), 인구(장기 트렌드)가 서로 다른 시간 스케일로 영향을 미친다. 이를
  하나의 데이터 소스에서 얻을 수 없으므로, 성격이 다른 세 개의 공공 API를 조합한다.

  예측 결과는 그 자체가 목적이 아니다. 실제 운영자가 필요한 것은 "예비력을 추가로
  확보할 것인가, 수요반응을 발동할 것인가, 현상 유지할 것인가"라는 행동 결정이다.
  Bayesian Expected Loss는 예측의 불확실성을 그대로 유지한 채 손실 관점에서 최적
  전략을 도출하기 위해 사용한다.

  어떤 API나 ML 모델이 실패해도 파이프라인이 멈추지 않도록 각 단계에 fallback을
  설계했다. 사용자에게는 항상 리포트가 반환되고, errors 필드에 경로 기록이 남는다.

[ 데이터 출처 및 활용 목적 ]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  전기수요에 영향을 주는 변수는 크게 세 가지 성격으로 나뉜다.

  ① 단기 변동 변수 — 수일 단위로 바뀌며 냉난방 부하를 직접 결정한다.
     → 기상청 단기예보 API로 수집

  ② 과거 시계열 — 실제 수요 패턴(요일 효과, 계절성, 추세)을 학습하는 데 사용한다.
     → 전력거래소 전국 수요 API로 수집, 지역 비중으로 배분

  ③ 느린 구조 변수 — 연 단위로 바뀌며 지역 기저수요 수준을 결정한다.
     → 통계청 KOSIS (GRDP, 인구)로 수집

  아래에 각 출처별로 수집 내용과 모델 내 역할을 상세히 서술한다.

[ STEP 1 ]  collect_node  —  공공데이터 수집
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  목적
    세 개의 공공 API를 호출해 서로 다른 시간 스케일의 원시 데이터를 수집하고
    ElecState에 적재한다. API 응답 실패는 collector 내부에서 흡수되며, 파이프라인은
    항상 다음 단계로 진행된다.

  공통 처리 메커니즘 (BaseCollector)
    ① settings.USE_MOCK=True 이거나 API 키가 없으면 즉시 mock()으로 분기
    ② 키가 있으면 fetch() 시도 → 성공 시 TTL 300초 메모리 캐시에 저장
    ③ HTTP 오류·파싱 실패 등 예외 발생 시 mock()으로 자동 fallback
    ④ 결과는 항상 { collector, source, data } 형태로 래핑하여 반환
       source 값: "api" | "mock(forced)" | "mock(fallback:오류명)"

[ STEP 2 ]  predict_node  —  ML 수요 예측
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  목적
    STEP 1에서 수집한 세 출처의 데이터를 통합 피처 테이블로 변환한 뒤, 향후 N일의
    전기수요(MWh)를 예측하고 95% 신뢰구간을 함께 산출한다.

  피처 엔지니어링 (_build_frame)
    과거 60일 power.history의 각 날짜에 대해 아래 피처를 계산한다.
    날씨 데이터는 미래 위주이므로 과거에는 연중일(doy) 기반 sine 곡선으로 대체한다.

    ┌────────────────┬─────────────────┬──────────────────────────────┐
    │ 피처           │ 출처            │ 산출 방식                    │
    ├────────────────┼─────────────────┼──────────────────────────────┤
    │ dow            │ date 파싱       │ 요일 (0=월 ~ 6=일)           │
    │ doy            │ date 파싱       │ 연중 일수 (1~365)            │
    │ temp_avg       │ weather/sine    │ 일 평균기온 (℃)              │
    │ cooling_load   │ temp_avg 파생   │ max(temp_avg - 24, 0)        │
    │ heating_load   │ temp_avg 파생   │ 저온 패널티 합산             │
    │ grdp           │ indpop          │ GRDP (조원)                  │
    │ ind_sales      │ indpop          │ 산업매출지수                 │
    │ pop            │ indpop          │ 인구 (천명)                  │
    │ demand_mwh     │ power.history   │ 학습 타깃                    │
    └────────────────┴─────────────────┴──────────────────────────────┘

  미래 피처 구성
    예측 대상 날짜(오늘+1 ~ 오늘+N)에 대해 weather.records의 실제 기온을 우선
    사용하고, 없으면 sine 프록시로 대체한다. GRDP·인구는 동일 값을 broadcast.

  예측 엔진 선택 로직
    [1순위] AutoGluon TabularPredictor
      - 조건: autogluon 설치됨 AND 학습 데이터 ≥ 20행
      - 학습: presets="medium_quality", time_limit=30s
      - 신뢰구간: 학습 잔차 σ → 예측값 ± 1.96σ (95% CI)
      - 피처중요도: feature_importance() 자동 산출

    [2순위] 선형추세 + 기온 민감도 Fallback
      - 조건: AutoGluon 미설치 또는 데이터 부족 또는 학습 실패
      - 방법: OLS 선형회귀로 slope·intercept 추정
              → trend × 요일계수 × 기온보정(냉난방 1도당 ±0.8%)
      - 피처중요도: 고정값 {temp:45%, trend:30%, weekday:15%, industry:10%}

  State 갱신 (출력)
    state["forecast"] = {
        region, horizon_days,
        method        : "autogluon" | "fallback_linear" | "failed",
        predictions   : [ {date, demand_mwh, lower, upper} × N ],
        feature_importance : { 피처명: 중요도 }
    }

[ STEP 3 ]  decision_node  —  Bayesian 기대손실 의사결정
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  목적
    예측 수요(forecast)와 과거 수요 분포(power.history)를 결합하여 "내일이 피크
    상태인가?"라는 이진 판단 문제를 Bayesian 추론으로 해결하고, 3가지 운영 전략의
    기대손실을 비교해 최적 전략을 권고한다.

  왜 Bayesian인가
    예측값은 항상 불확실성을 가진다. 단순히 "예측 수요가 높다"고 바로 예비력을
    늘리면 불필요한 비용이 발생하고, "낮다"고 방치하면 피크 시 정전 페널티가 발생
    한다. Bayesian Expected Loss는 이 불확실성을 확률로 유지한 채 손실의 기댓값을
    최소화하는 전략을 선택한다는 점에서 합리적이다.

  추론 과정

    ① 사전확률 P(peak) — 계절성 반영
       여름(7~8월) / 겨울(12~1월) : 0.45   (역사적으로 전기수요 급증 시기)
       봄가을 경계 (6·9·2월)       : 0.30
       그 외 (봄·가을)              : 0.15

    ② 우도(Likelihood) — 예측 수요의 z-score 기반
       hist_mean, hist_sd = 과거 60일 history의 평균·표준편차
       peak_pred          = forecast.predictions 중 최대 수요

       z = (peak_pred - hist_mean) / hist_sd
         → z가 클수록 이번 수요가 과거 분포에서 얼마나 이상적으로 높은지를 나타냄

       like_peak   = sigmoid(z - 1.0)   (z > 1에서 급증)
       like_normal = 1 - like_peak

    ③ 사후확률 — Bayes 정리
       posterior = (like_peak × prior) /
                   (like_peak × prior + like_normal × (1 - prior))

    ④ 기대손실 산출
       손실표 L(전략, 상태) — 상대 비용 단위:

       전략              │ 피크 발생  │ 정상 상태 │ 해석
       ──────────────────────────────────────────────────────────
       hold (무조치)     │   100.0    │    0.0    │ 피크 시 대응 불가 → 최대 손실
       demand_resp (DR)  │    35.0    │   12.0    │ DR 발동 비용 감수, 피크 리스크 완화
       ramp_supply (발전)│    20.0    │   30.0    │ 예비력 확보 비용 高, 피크엔 안전

       E[L | action] = P(peak) × L(action, peak)
                     + P(normal) × L(action, normal)
       → 3개 전략 중 E[L] 최솟값 전략을 recommended로 선택

  결정 논리 예시
    posterior=0.3 (봄가을, z=0.5) → E[L|hold]=30, E[L|DR]=17.5, E[L|ramp]=23
    → 권고: demand_resp (비용은 들지만 hold보다 기대손실 낮음)

    posterior=0.6 (여름, z=2.0)   → E[L|hold]=60, E[L|DR]=25.8, E[L|ramp]=24
    → 권고: ramp_supply (높은 피크 확률에서 예비력 확보가 최적)

  State 갱신 (출력)
    state["decision"] = {
        posterior_peak  : P(peak | 증거)
        expected_loss   : { hold, demand_resp, ramp_supply }
        recommended     : 기대손실 최소 전략명
        rationale       : 추론 근거 문자열 (사전→z→사후→최적전략)
    }

[ STEP 4 ]  strategy_node  —  운영 전략 생성 (Gemini LLM / 규칙 기반)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  목적
    Bayesian 의사결정 결과를 맥락으로 삼아, 실제 운영자가 선택할 수 있는 A/B 두
    가지 행동 옵션과 지역 특성을 반영한 SWOT 분석을 생성한다.

  이 단계가 필요한 이유
    decision_node는 기대손실이 최소인 전략을 수치로 선택하지만, 실제 운영에서는
    "어떤 순서로, 어떤 자원을 동원해서" 수행할지가 중요하다. strategy_node는
    수치 결정 결과를 구체적 행동 시나리오로 번역하는 역할을 한다.

  처리 우선순위

    [1순위] Gemini LLM (gemini-2.5-flash)
      - 입력: ElecState 전체 직렬화 → build_user_prompt(state)
      - 시스템 프롬프트: SWOT 기반 전략 생성 지시
      - 출력: JSON 파싱 → { summary, option_a, option_b, swot }
      - LLM이 추가 가치를 제공하는 부분:
          - 지역 산업 특성과 계절 정보를 조합한 맥락 있는 전략 서술
          - 수치로 표현하기 어려운 운영 리스크와 기회 포착
          - Bayesian 권고와 상이한 시나리오를 option_b로 제시하여 대안 확보

    [2순위] 규칙 기반 Fallback (LLM 없거나 실패 시)
      posterior_peak 임계값 0.5로 분기:

      [고위험: posterior ≥ 0.5]
        option_a: 예비력 발전 우선 기동 + DR 사전 공지
        option_b: 대형 산업수요 DR 계약 발동 + 비핵심 부하 시간이동

      [저위험: posterior < 0.5]
        option_a: 정규 급전 유지 + 실시간 모니터링
        option_b: 부분 예비력 대기 + 기상 급변 알림 설정

      SWOT (지역 산업 규모 반영):
        GRDP > 100조: S = "고부가 산업 밀집으로 안정적 기저수요"
        GRDP ≤ 100조: S = "수요 변동성 낮음"
        W (공통): "냉난방 피크 민감도"
        O (공통): "DR 자원·재생E 연계 확대"
        T (공통): "기상 이변에 따른 급격한 피크"

  State 갱신 (출력)
    state["strategy"] = {
        summary   : 상황 요약 문자열
        option_a  : { name, actions: [...], tradeoff }
        option_b  : { name, actions: [...], tradeoff }
        swot      : { strength, weakness, opportunity, threat }
    }

  목적
    ElecState의 모든 처리 결과를 하나의 Markdown 리포트로 조합한다.
    각 단계의 데이터 출처(sources)를 명시하여 결과의 신뢰성과 추적성을 보장한다.

  리포트 구성
    # {region} 전기수요 예측 리포트
    _생성: YYYY-MM-DD HH:MM_

    - 예측 방법: autogluon | fallback_linear
    - 데이터 출처: { weather, power, industry_population }

    ## 일자별 예측 수요 (MWh)
    - YYYY-MM-DD: X,XXX MWh (95% CI A ~ B)  ×N일

    ## 피크 리스크 의사결정 (Bayesian Expected Loss)

    ## 운영 전략 A/B
    - option_a / option_b 행동 목록 + 트레이드오프

    ## SWOT
    - S / W / O / T

    [오류 발생 시]
    > ⚠️ degraded paths: [에러 목록]

  State 갱신
    state["report"] = 완성된 Markdown 리포트 문자열
